## Context Switch

在现代操作系统中，Context Switch 分为两个层级：**进程间切换**和**线程间切换**。虽然它们都叫 CS，但代价和动作大不相同。



### 进程切换 vs 线程切换：代价的鸿沟

虽然都叫 CS，但**线程切换**比**进程切换**快得多，原因就在你刚才问的“虚拟地址空间”里：

#### **进程间切换（重装系统）**
1.  **保存全套寄存器**。
2.  **切换虚拟地址空间**：这是最耗时的。CPU 需要修改 `CR3` 寄存器（页表基址），这意味着**页表缓存（TLB）会全部失效**。
3.  **刷新缓存**：由于地址空间变了，CPU 的一级、二级缓存（L1/L2 Cache）命中率会暴跌。

#### **线程间切换（换个工人）**
1.  **保存全套寄存器**。
2.  **不切换虚拟地址空间**：因为同一个进程的线程共享同一套地址空间（代码段、堆、全局变量）。
3.  **保留 TLB 和 Cache**：CPU 访问内存的效率依然很高。

### 重新审视虚拟地址空间布局

我们把线程并发的动作放进你刚才提到的“从上到下”的内存布局里看：

| 区域 | 线程并发时的变化 |
| :--- | :--- |
| **内核空间** | **CS 发生地**。内核在此保存线程的 TCB（线程控制块）。 |
| **栈 (Stack)** | **每个线程独有一份**。切换时，指针从线程 A 的栈跳到线程 B 的栈。 |
| **共享库/堆/数据段** | **所有线程共享**。切换时，这部分不动，线程 B 能直接读线程 A 改过的数据。 |
| **代码段 (Text)** | **共享**。所有线程都在这同一片区域取指令。 |


### 什么时候会发生 CS？

在 Linux 系统中，只要看到以下情况，都在发生 Context Switch：

1.  **抢占式调度**：线程的时间片用完了，被迫让出 CPU。
2.  **阻塞/等待**：线程在等 IO（读磁盘、等网络）、等锁（Mutex），主动进入睡眠。
3.  **中断处理**：硬件发出中断请求，内核强制切换上下文去处理紧急事务。

### 💡 核心结论
* **函数调用**：不换工人，不进内核，不叫 CS。
* **线程切换**：换工人，进内核，**叫轻量级 CS**（不换地址空间）。
* **进程切换**：换工厂，进内核，**叫重量级 CS**（换全套地址空间）。
