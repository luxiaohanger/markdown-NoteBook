# 原子性

在多线程底层世界里，**原子性**是防止崩溃和逻辑错误的最后防线。

原子操作之间空间的属于可竞争空间，满足条件的线程都有资格抢占资源，并发设计的核心之一就是允许合理的竞争出现，提升效率，同时保证必要的原子性，维护秩序。

-----

## 原子性与层级

### 1. 硬件/指令层级（绝对原子）
这是所有同步机制的基石。这些操作由 CPU 硬件保证，不会被任何中断或切换打断。

* **CAS (Compare-And-Swap):** 用户态实现“快路径”的核心。它原子的比较内存值并交换。
* **Atomic Increment/Decrement:** 比如 C++ 中的 `std::atomic` 或 Java 的 `AtomicInteger`。
* **特性：** 它们是**非阻塞**的，执行速度极快（几个纳秒）。

### 2. 系统调用层级：Futex（内核保证的原子性）
`futex_wait` 的内部逻辑是**复合原子性**的。

* **`futex_wait(uaddr, val)`：** 
   * **它是原子的吗？** 是的。内核通过加锁（Hash Bucket Lock）保证了“读取 `uaddr` 的值”与“将线程放入等待队列”这两个动作之间，没有任何其他线程能修改这个 `uaddr` 的状态。
    * **意义：** 它解决了“检查”与“睡眠”之间的时间差问题，彻底杜绝了**丢失唤醒**。
* **`futex_wake(uaddr, count)`：**
    * **它是原子的吗？** 是的。它原子的从等待队列中移除指定数量的线程并标记为就绪。

### 3. 高层抽象层级：Mutex 和 Cond Var（逻辑原子性）
到了这个层级，原子性的定义发生了一点变化。

#### 锁 (Mutex)
* **`pthread_mutex_lock`：**
    * **逻辑原子性：** 从外部看，它保证了“获取锁”这个动作是原子的——要么拿到了，要么没拿到去排队。
    * **实现：** 它内部是由 **用户态 CAS（原子指令）** + **`futex_wait`（原子系统调用）** 组合而成的。
* **`pthread_mutex_unlock`：** 同上，它原子的释放所有权并可能触发唤醒。

#### 条件变量 (Condition Variable) —— 最特殊的一个
* **`pthread_cond_wait(cond, mutex)`：**
    * **关键点：** 这个 API 内部包含了**两个动作的原子切换**。
    * **原子性表现：** 它会**原子地释放锁并进入睡眠**。
    * **为什么必须这样？** 如果先释放锁，再调 `futex_wait`，中间的空隙可能导致别的线程改了条件并发出信号，导致信号丢失。所以，`cond_wait` 必须由内核保证“释放锁”和“挂起自己”之间是不可分割的。

### 总结表：谁是原子的？

| API / 抽象 | 是否原子 | 原子性的保障者 | 解决的问题 |
| :--- | :--- | :--- | :--- |
| **CAS / Atomic** | **是** | CPU 硬件 (总线锁/缓存锁) | 基础数据竞争 |
| **`futex_wait`** | **是** | 内核 (Hash Bucket Lock) | **丢失唤醒** (检查与睡眠的空隙) |
| **`mutex_lock`** | **逻辑原子** | CAS + Futex | 互斥访问资源 |
| **`cond_wait`** | **复合原子** | 内核 | **信号丢失** (释放锁与睡眠的空隙) |



### 💡 深度理解：
你可能会发现，所谓的“原子性”其实是一层套一层的。
* 硬件原子性是为了保护**内存数据**。
* Futex 原子性是为了保护**睡眠状态**。
* Mutex 原子性是为了保护**业务逻辑**。

当你觉得某个操作“不是原子”的时候，通常是因为你发现两个操作之间存在**时空裂缝**，而这个裂缝可以被其他线程插足。所有的同步 API，本质上都是在试图用更大的“原子块”来封堵这些裂缝。

## 什么是原子性？

**原子性（Atomicity）并不是指“让时间停止”或“禁止一切竞争”，而是指“操作在逻辑上不可分割”以及“状态切换的瞬时性”。**


### 1. 原子性 $\neq$ 流程封闭
你看到的 `lock` 或 `unlock` 流程，其实像是一连串由**原子锚点**连接起来的虚线。
* **锚点（Atomic Anchors）：** 比如 `CAS` 指令或 `futex_wait` 系统调用。在这些点上，硬件或内核保证了操作的绝对独立。
* **虚线（The Gaps）：** 在这些锚点之间，系统是“开放”的。其他线程可以插队、可以修改值、可以抢夺 CPU。

**原子性的真正意义是：** 即使你在虚线部分被插队了，当你回到下一个“锚点”时，系统有能力检测到这种变化（比如 Futex 的预期值校验），并让你做出正确的后续反应（重试或报错），从而**维持逻辑的连续性**。

### 2. 原子性 = 状态的“量子跳跃”
在并发世界里，原子性其实是在定义**状态切换**。
* **非原子操作：** 状态从 $A$ 变到 $B$ 经历了一个模糊的过程。就像推一扇沉重的旋转门，门半开半闭时，谁都能挤进来。
* **原子操作：** 状态从 $A$ 直接“闪现”到 $B$。对于外界观察者来说，这个变量要么是 $A$，要么是 $B$，不存在“中间态”。


### 3. 原子性是用来“管理”竞争，而不是“消除”竞争
你提到的“数据允许竞争”非常关键。原子操作存在的目的，正是为了**在激烈的竞争中维持秩序**。

* **没有原子性：** 两个线程同时改数据，结果可能是一个乱码（数据损坏）。
* **有了原子性：** 两个线程同时改数据，**硬件会强制排队**。最终结果要么是线程 1 的值，要么是线程 2 的值，绝不会是两者的混合。

## 分层原子性
**lock() 和 unlock（）在宏观逻辑上，它是原子的；但在微观实现上，它是一个“混合原子操作”。**

他们是**硬件原子指令**和**内核原子系统调用**的分层组合。

### 1. LOCK

我们可以把 `lock` 的过程拆解为两个阶段，你会发现它也不是一个从头到尾被“锁死”的单一原子块。

#### 第一阶段：用户态快路径 (Fast Path) —— **绝对原子**
当线程尝试加锁时，它首先执行一条硬件级别的原子指令（如 `CAS` 或 `Exchange`）。
* **操作：** `CAS(lock_addr, 0, 1)`。
* **原子性：** 由 CPU 硬件保证。在这一纳秒级的时间内，没有任何其他 CPU 核心可以修改这个地址。
* **结果：** 如果返回成功，线程直接拿到锁。**此时，对于这个线程来说，`lock` 是完全原子的。**

#### 第二阶段：内核态慢路径 (Slow Path) —— **复合原子**
如果第一阶段失败（发现锁已经是 `1` 了），它就必须调用 `futex_wait`。
* **操作：** `syscall(SYS_futex, uaddr, FUTEX_WAIT, 1)`。
* **原子性：** 正如我们之前讨论的，`futex_wait` 在内核内部是原子的（检查值 + 挂起）。
* **结果：** 线程进入睡眠。


> 如果我们将“第一阶段”和“第二阶段”合起来看，中间其实存在一个**非原子性的间隙**：

1.  **线程 A** 执行 CAS 发现锁被占用（失败）。
2.  **间隙发生：** 在线程 A 准备调用 `futex_wait` 之前，原有的锁持有者突然 **解锁了** 并修改值为 `0`。
3.  **线程 A** 进入内核执行 `futex_wait(uaddr, 1)`。

**这时，Futex 的原子校验就起作用了：**
内核发现 `uaddr` 的值已经是 `0` 了，与预期的 `1` 不符，于是 `futex_wait` 拒绝让线程 A 睡眠并直接返回。线程 A 回到用户态再次重试 CAS。

> **结论：** 这种设计允许在“想睡”和“真睡”之间插入其他线程的操作，但通过内核的**原子校验**确保了逻辑的正确性。

### 2. UNLOCK 

在 Linux 的 `pthread_mutex_unlock` 实现中，它通常包含这两个独立的步骤：

* **动作 A（用户态原子指令）：** `atomic_store(lock_addr, 0)`。这个动作瞬间完成，锁在这一刻就已经“开”了。
* **动作 B（内核态系统调用）：** `futex(uaddr, FUTEX_WAKE, 1)`。内核去队列里叫醒一个人。


> 如果 **动作 A** 做了，但在 **动作 B** 执行前，CPU 切换走了，或者发生了别的事情怎么办？

这正是 Futex 设计的高明之处：**锁的所有权完全由用户态的值决定。**

1.  **锁已经开了：** 只要 **动作 A** 完成，用户态的值变成了 `0`。此时，如果有任何**新线程**（不在等待队列里的）冲过来加锁，它可以通过“快路径”（CAS）瞬间抢到锁并开始工作。
2.  **唤醒只是“延迟”了：** 即便执行 **动作 B**（唤醒）慢了一点点，结果也只是让正在睡觉的线程多睡了几个微秒。它最终还是会被叫醒。
3.  **竞争是公平的：** 被叫醒的线程回到用户态后，会重新检查锁的状态。如果锁被刚才那个“冲过来的新线程”抢走了，它就继续调用 `futex_wait` 睡回去。

### 3. COND_WAIT
**`pthread_cond_wait()` 在逻辑上表现为原子性，但在底层实现（内核态）也并非一个单一的、不可分割的硬件原子操作。**


当你调用 `pthread_cond_wait(cond, mutex)` 时，它必须完成两个动作：
1. **释放锁** (`pthread_mutex_unlock`)
2. **挂起线程** (进入等待队列)

如果这两个动作之间有缝隙，就会出现所谓的“信号丢失” (Lost Wakeup) 风险：线程刚放锁还没睡着，另一个线程就把 `signal` 发出来了，等第一个线程真正去睡时，已经没人叫醒它了。

和上述两个函数一样，系统在内核层面为这种原子性缺失设置了兜底机制，或者说系统在内核层面是原子的。为了防止在“进入内核准备睡觉”的过程中错过 `signal`，`futex_wait` 采用了 **预期值校验机制**。


#### `futex_wait` 中的 `uaddr` 指向什么？

在 `pthread_cond_wait` 的上下文中，`uaddr`指向`pthread_cond_t`内部的一个**计数器（Counter）**。


* **作用：** 这个值本质上是一个**序列号或等待计数器**。
* **流程：**
    1. 当 `pthread_cond_signal` 或 `broadcast` 被调用时，内核会递增这个 `uaddr` 指向的数值。
    2. 当 `pthread_cond_wait` 调用时，它会记下当时的 `val` 并传给内核。
    3. 内核对比 `uaddr` 的当前值和 `val`，只要两者不一致，就证明自你打算睡去之后，已经有 `signal` 发生了。

> 这三者都属于涉及到用户态和内核态相互切换的系统调用，系统不保证切换空隙的原子性，但是系统保证了切换到内核态的二次校验（主要体现在futex_wait）保证了逻辑上的“原子性”

## futex 的原子性

**在 `futex_wait` 的系统调用内部，通过内核锁保证了“检查值”和“挂起线程”之间是绝对没有数据竞争的。**

如果这两个动作之间有空隙，就会发生“丢失唤醒”。为了封堵这个缝隙，Linux 内核在实现 `futex_wait` 时使用了一套精妙的**二级锁机制**。


### 1. 内核是如何封堵这个缝隙的？

当你从用户态进入内核态执行 `futex_wait(uaddr, expect_val)` 后，内核并不会直接读取数据，而是先进行“阵地管理”：

1.  **查找哈希桶（Hash Bucket）：** 内核维护着一张全局的 Futex 等待表。它会根据用户传入的地址 `uaddr` 计算一个哈希值，找到对应的“哈希桶”。
2.  **获取桶锁（Bucket Spinlock）：** **这是关键。** 内核会锁住这个桶。一旦锁住，任何试图对同一个地址执行 `futex_wake` 或其他 `futex_wait` 的线程都会在此阻塞。
3.  **最后一次检查（The Critical Check）：** 在持有桶锁的状态下，内核去读取用户态地址 `uaddr` 的当前值。
    * **如果当前值 != `expect_val`：** 说明在你进入内核的这一瞬间，有人改了数据。内核立即释放桶锁，并返回错误（`EAGAIN`）。
    * **如果当前值 == `expect_val`：** 说明条件依然成立。内核将你的线程对象（Task Struct）放入这个桶的等待队列中。
4.  **设置睡眠状态：** 将线程标记为睡眠，然后释放桶锁。


### 2. 为什么这样就不会有竞争？

因为 **`futex_wake` 也要拿同一把桶锁**。

* 如果**生产者**想要唤醒**消费者**，它必须调用 `futex_wake`。
* `futex_wake` 进入内核后，第一件事也是根据 `uaddr` 找到同一个哈希桶并尝试获取那把**桶锁**。
* 由于**消费者**此时正拿着锁在做“最后一次检查”，**生产者**必须等待。

这保证了：要么唤醒发生在检查之前，要么发生在检查并入队之后，绝不会发生在“检查完还没入队”的那个尴尬瞬间。


### 3. 这里的“原子性”边界在哪里？

我们需要区分两种竞争：

#### A. 用户态变量的竞争（允许发生）
在用户态，你的代码可以随时修改 `uaddr` 指向的那个整数。Futex 并不禁止这种竞争，它只是通过 `expect_val` 来**检测**这种竞争。如果检测到竞争发生，它就拒绝让你睡觉。

#### B. 内核同步状态的竞争（绝对禁止）
“读用户值”和“把线程放进队里”这两个动作，在内核层面通过桶锁（Spinlock）变成了一个**封闭的原子块**。


### 4. 深度总结：Futex 的“双保险”

我们可以把 `futex_wait` 的逻辑简化为一段伪代码

```c
// 进入内核态
spin_lock(&hash_bucket->lock); // 1. 锁住阵地，禁止别人 wake 
if (*uaddr == expect_val) {    // 2. 在锁的保护下看一眼
    enqueue(current_thread);   // 3. 如果没变，立刻入队（占位）
    spin_unlock(&hash_bucket->lock);
    schedule();                // 4. 放心睡觉
} else {
    spin_unlock(&hash_bucket->lock);
    return -EAGAIN;            // 5. 有人动过了，回去重试
}
```

> **原子性其实是靠更底层的锁来支撑**
