# Semaphores
POSIX 信号量（POSIX Semaphores）是 Linux/Unix 系统中用于多线程或多进程间**同步**与**互斥**的一种机制。

简单来说，它就像是一个**计数器**，用来管理某种公共资源的可用数量。

------
## 总览

### 1. 核心作用：它能解决什么问题？

信号量的主要作用可以概括为：**资源计数**和**行为同步**。

* **互斥（Mutual Exclusion）：** 确保同一时刻只有一个线程能访问某个特定资源（此时信号量初始值为 1，等同于锁）。
* **资源管理（Resource Management）：** 当你有多个相同的资源（比如 5 台打印机）时，信号量可以跟踪还有多少台可用。
* **同步（Synchronization）：** 控制线程执行的先后顺序。例如，线程 A 必须在线程 B 完成某项任务后才能继续，B 完成后通过信号量给 A 发个“信号”。

> **形象比喻：**
> 信号量就像是一个停车场的**剩余车位显示屏**。
> * 当车进入时（`sem_wait`），显示屏数值减 1。如果数值为 0，想进的车必须在门口排队等待。
> * 当车离开时（`sem_post`），显示屏数值加 1，排在最前面的车可以进入。


### 2. POSIX 信号量的两种类型

根据使用范围，POSIX 信号量分为两类：

| 类型 | 函数前缀 | 存储位置 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **无名信号量 (Unnamed)** | `sem_init` | 内存中（全局/静态变量） | 同一进程内的线程间，或有亲缘关系的进程间（需在共享内存中）。 |
| **有名信号量 (Named)** | `sem_open` | 文件系统中（通常在 `/dev/shm`） | 任意进程间同步。 |


### 3. 使用语法（核心 API）

使用 POSIX 信号量需要包含头文件 `<semaphore.h>`，并在编译时链接线程库 `-lpthread`。

#### A. 无名信号量（线程间常用）

1.  **初始化**
    ```c
    int sem_init(sem_t *sem, int pshared, unsigned int value);
    ```
    * `sem`: 指向信号量对象的指针。
    * `pshared`: `0` 表示线程间共享，`非0` 表示进程间共享。
    * `value`: 信号量的初始值。

2.  **等待操作 (P 操作) - 减少计数值**
    ```c
    int sem_wait(sem_t *sem); // 如果值为0则阻塞，直到值大于0
    int sem_trywait(sem_t *sem); // 非阻塞版，如果为0直接返回错误
    ```

3.  **释放操作 (V 操作) - 增加计数值**
    ```c
    int sem_post(sem_t *sem); // 将信号量值加1，并唤醒正在等待的线程
    ```

4.  **销毁**
    ```c
    int sem_destroy(sem_t *sem);
    ```

#### B. 有名信号量（进程间常用）

1.  **打开/创建**
    ```c
    sem_t *sem_open(const char *name, int oflag, mode_t mode, unsigned int value);
    ```
    * `name`: 以 `/` 开头的名称（如 `"/my_sem"`）。

2.  **关闭与删除**
    ```c
    int sem_close(sem_t *sem); // 关闭连接
    int sem_unlink(const char *name); // 从系统中彻底删除
    ```

## thread safety
PV函数内部由内核态的锁保证原子性和线程安全，但是PV函数之间的用户态代码区没有必然的线程安全保证

- value 初始为 1 且所有线程中 PV 配对出现：
此时信号量相当于互斥锁，当某个线程 wait() 进入后，其他线程被阻挡在临界区外，直到 post() 释放；
如果所有线程遵守PV先后配对出现，就能在临界区实现mutex的功能；

- 但是如果线程直接post()，那样PV之间就没有安全可言

## 使用场景
虽然 Mutex + CV 在逻辑严密性上更胜一筹，但信号量（Semaphore）之所以长盛不衰，是因为它在某些场景下具有**天然的语义优势**和**极致的简洁性**。


### 1. 资源池管理（The Resource Pool）
这是信号量最本色的舞台。假设你有一个数据库连接池，里面只有 5 个连接。

* **信号量的做法：**
    初始化 `sem_t sem` 为 5。
    * 想用连接？`sem_wait(&sem)`。
    * 用完还回去？`sem_post(&sem)`。
    **优势：** 一行代码搞定限流。
* **Mutex + CV 的做法：**
    你需要一个 `int count = 5`，一把锁，一个条件变量。每次拿连接都要 `lock -> while(count==0) wait -> count-- -> unlock`。
    **对比：** 信号量将“数值检查”和“阻塞等待”原子化地封装在一起，代码量极小，逻辑非常直观。


### 2. 生产者-消费者模式（特别是环形缓冲区）
在处理固定大小的缓冲区时，信号量表现得异常优雅。

通常需要两个信号量：
1.  `sem_t empty` (初始值为 N)：代表空槽位的数量。
2.  `sem_t full` (初始值为 0)：代表已填充数据的数量。

* **生产者：** `wait(empty)` -> 放入数据 -> `post(full)`
* **消费者：** `wait(full)` -> 取出数据 -> `post(empty)`

**为什么优于 Mutex+CV？**
在 Mutex+CV 中，生产者唤醒消费者后，消费者还得抢锁、重新检查条件。而信号量利用其**“有状态”**的特性，直接完成了计数的增减，不需要额外的变量维护，且由于信号量没有所有权，生产者和消费者可以完全解耦地进行 V 操作。


### 3. 异步事件通知（Asynchronous Signaling）
这种场景下，信号量几乎是碾压 Mutex+CV 的存在。

**场景：** 线程 A 负责初始化硬件，线程 B 必须等待硬件就绪后才能开始工作。
* **信号量：** 初始值为 0。线程 B 直接 `wait`。线程 A 初始化完直接 `post`。
* **优势：**
    * **不怕错过信号：** 如果线程 A 初始化太快，在线程 B 还没运行到 `wait` 时就先 `post` 了，信号量的计数值会变成 1。等线程 B 运行到 `wait` 时，它会瞬间通过，不会阻塞。
    * **CV 的尴尬：** 条件变量是“无状态”的。如果线程 A 在 B `wait` 之前发了 `signal`，这个信号就丢了！线程 B 随后进入 `wait` 将永远无法被唤醒（除非你额外引入一个布尔变量和一把锁来记录状态）。

### 4. 信号处理函数（Signal Safety）
这是系统编程中的一个“硬核”优势。

在 Linux 中，很多函数是**非信号安全（Async-Signal-Unsafe）**的。例如，你**绝对不能**在信号处理函数（Signal Handler，如处理 `Ctrl+C` 的函数）里调用 `pthread_mutex_lock`，因为如果主程序正持有这把锁，就会导致死锁。

但是，`sem_post` 是**信号安全**的。
* **场景：** 当按下 `Ctrl+C` 时，信号处理函数捕获到信号，仅仅执行一个 `sem_post(&sem)`，主循环里的 `sem_wait` 就会被唤醒去执行清理工作。
* **结论：** 这是在这种极端环境下，唯一能安全使用的同步原语。


### 5. 跨进程同步（IPC）
虽然 Mutex 也可以通过设置 `PTHREAD_PROCESS_SHARED` 属性来实现跨进程，但配置繁琐。

* **有名信号量（Named Semaphore）：** 只需要一个字符串名称（如 `"/my_gate"`），两个互不相关的进程就可以通过这个名字找到同一个信号量并进行同步。
* **优势：** 像文件一样好用，不需要父子进程关系，也不需要复杂的共享内存设置。


### 总结：什么时候该选信号量？

| 场景 | 选信号量的理由 |
| :--- | :--- |
| **限流/计数** | 逻辑最简，不需要额外维护 `int` 变量。 |
| **异步通知** | 信号不会丢失，不要求等待者先“就位”。 |
| **中断/信号处理** | 它是异步信号安全的，Mutex 则不是。 |
| **解耦协作** | `wait` 和 `post` 不在同一个线程或进程，不需要所有权约束。 |
