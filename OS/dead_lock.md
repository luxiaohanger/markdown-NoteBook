## 死锁 (Deadlock)
在并发编程中，**死锁 (Deadlock)** 就像是两个人在狭窄的过道相遇，每个人都坚持让对方先走，结果谁也动弹不了。

要避免这种情况，我们通常从破坏死锁发生的 **四个必要条件**（互斥、占有且等待、不可剥夺、循环等待）入手。以下是几种主流且实用的策略：

### 1. 破坏“循环等待”：顺序加锁 (Lock Ordering)
这是最常用且最有效的策略。死锁通常是因为线程 A 锁住了资源 1 等待资源 2，而线程 B 锁住了资源 2 等待资源 1。

* **策略：** 规定所有线程必须按照**相同的顺序**获取锁。
* **示例：** 如果你需要同时操作账户 A 和账户 B，规定总是先锁定 ID 较小的账户，再锁定 ID 较大的账户。



### 2. 破坏“占有且等待”：一次性申请 (Request All at Once)
线程在开始执行前，必须一次性申请到所有需要的资源。如果有一部分资源申请不到，就不占有任何资源。

* **策略：** 使用一个全局的“资源管理器”或“信号量”。线程先向管理器申请所有锁，只有全部满足时才开始执行。
* **优点：** 简单直接。
* **缺点：** 资源利用率低，可能导致某些线程迟迟无法开始（饥饿现象）。

### 3. 破坏“不可剥夺”：超时放弃 (Lock Timing)
如果一个线程尝试获取锁失败，它不应该无休止地等下去。

* **策略：** 使用带有超时的锁机制（如 Java 中的 `tryLock(timeout)`）。
* **动作：** 如果在规定时间内没有获得锁，线程会释放它已经持有的所有锁，然后等待一段随机时间后再重试。这能有效打破僵局。

### 4. 银行家算法 (Banker's Algorithm)
这是一种预防死锁的经典算法，常用于操作系统层面。

* **原理：** 在分配资源前，系统会先计算这次分配是否会导致系统进入“不安全状态”。如果可能导致死锁，系统就不予分配，让线程等待。
* **核心公式：** $$Available \ge Need(i)$$
    系统通过模拟分配后的状态，确保总能找到一条路径让所有进程安全完成。



### 5. 减少锁的粒度与可见性
有时候，问题的根源在于我们“锁得太多”或“锁得太久”。

* **使用无锁编程：** 尽量使用原子变量（如 `AtomicInteger`），利用 CAS (Compare And Swap) 操作来规避传统的锁机制。
* **缩小锁范围：** 只在必须保护的代码块上加锁，尽快释放。
* **读写分离：** 使用 `ReadWriteLock`。多个读者可以同时访问，只有写者才会互斥，这减少了冲突概率。


### 总结与建议
在实际开发中，**“预防为主，监控为辅”**：
1.  **首选顺序加锁**，这是成本最低的方案。
2.  **强制设置超时**，给程序留一个“撤退”的机会。
3.  **使用分析工具**，如 Java 的 `jstack` 或可视化工具来检测运行时的死锁轨迹。
