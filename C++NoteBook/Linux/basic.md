## listen connect accept

进程通过socket api通信，而tcp/udp传输层过程对程序员不可见，程序员只需要操作socket，具体来说是操作socket的文件描述符fd。

tcp的请求端和服务端的socket逻辑不完全一样：
- server需要对一个接收端口进行监听，此时的fd是一个不完整的，因为tcp socket 五元组缺少目的ip和port，（此时叫做监听套接字）当accept从全连接队列接收一个请求时，从原来的fd生成一个全新的fd，这是一个完整的socket，此后进程只需要操作此fd即可进行通信。

- client需要主动发起连接，因此先建立一个socket，但是不需要填写本地信息，OS会负责挑选空余的port进行隐式绑定；client需要填写目的信息，并使用connect向目的端口发起连接请求，成功后返回原fd

### 1. listen：开启监听门卫

`listen` 函数的作用是将一个**主动套接字**（Active Socket）转换成**被动套接字**（Passive Socket），告诉内核：这个端口已经准备好接收外部连接了。

*   **函数原型：** `int listen(int sockfd, int backlog);`
*   **参数详解：**
    *   `sockfd` (int): 由 `socket()` 创建的文件描述符，且必须已经通过 `bind()` 绑定了 IP 和端口。
    *   `backlog` (int): 提示内核为该监听套接字排队的最大连接数。
*   **工作原理：**
    内核为每个监听套接字维护两个队列：
    1.  **半连接队列（SYN Queue）：** 接收到客户端 SYN 包，正在进行三次握手的连接。
    2.  **已完成连接队列（Accept Queue）：** 已完成三次握手，等待进程调 `accept` 取走的连接。
*   **抽象目的：** 将“等待连接”这一高并发、高频的网络底层操作由内核托管，让应用层不需要处理复杂的握手细节。


### 2. connect：发起主动进攻

`connect` 是客户端调用的函数，用于向服务器发起 TCP 三次握手。

*   **函数原型：** `int connect(int sockfd, const struct sockaddr *addr, socklen_t addrlen);`
*   **参数详解：**
    *   `sockfd` (int): 客户端创建的套接字。
    *   `addr` (struct sockaddr*): 包含目标服务器 IP 和端口的结构体指针。
    *   `addrlen`: 结构体的长度。
*   **返回值：** 成功返回 0，失败返回 -1 并设置 `errno`（如 `ECONNREFUSED` 表示目标端口未监听）。
*   **工作原理：** 
    调用 `connect` 会触发内核发送第一个 **SYN** 包。只有当三次握手完全成功后，`connect` 才会返回。
*   **使用场景：** 典型的 TCP 客户端逻辑。


### 3. accept：获取连接成果

`accept` 并不是发起连接，而是从内核的**已完成连接队列**中“摘取”一个已经建立好的连接。

*   **函数原型：** `int accept(int sockfd, struct sockaddr *addr, socklen_t *addrlen);`
*   **返回值 (关键细节)：**
    *   成功时返回一个**全新的文件描述符**，专门用于与该客户端进行数据通信。
    *   原本的 `sockfd`（监听套接字）保持不变，继续等待其他新连接。
*   **参数详解：**
    *   `addr` 和 `addrlen`: 这是**输出参数**。内核会填入客户端的 IP 地址和端口信息。如果你不关心客户端是谁，可以传 `NULL`。
*   **逻辑与原理：**
    如果队列为空且套接字是**阻塞**的，`accept` 会一直睡眠直到有新连接到来。它是典型的“生产者-消费者”模型中的消费者。


### 核心关联：TCP 三次握手与函数的对应关系

为了深入理解，我们需要看这三个函数如何与 TCP 状态机联动：

1.  **SYN (客户端 -> 服务器):** 客户端调用 `connect()` 发出。此时服务器必须处于 `LISTEN` 状态（即已调用 `listen()`）。
2.  **SYN + ACK (服务器 -> 客户端):** 内核自动响应，此时连接进入服务器的**半连接队列**。
3.  **ACK (客户端 -> 服务器):** 客户端 `connect()` 收到响应并发送最后一包，`connect()` 成功返回。
4.  **连接入队:** 服务器内核收到最后一包 ACK，将连接移入**已完成连接队列**。
5.  **应用层获取:** 服务器进程调用 `accept()`，从队列头部取出一个连接，生成通信套接字。


### 总结：使用场景与搭配

| 函数 | 调用方 | 核心职责 | 搭配逻辑 |
| :--- | :--- | :--- | :--- |
| **listen** | 服务端 | 划定“待机”区域 | `socket` -> `bind` -> **listen** |
| **connect** | 客户端 | 主动建立连接 | `socket` -> **connect** -> `send/recv` |
| **accept** | 服务端 | 领走已就绪的客人 | **listen** -> **accept** -> `read/write` |
