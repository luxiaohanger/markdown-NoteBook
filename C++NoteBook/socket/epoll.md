## 核心作用与原理

### 1. 作用

`epoll` 是 Linux 内核为处理大批量文件描述符（FD）而作了改进的 **I/O 多路复用机制**。它能够监控多个文件描述符，当某个描述符就绪（可读、可写或异常）时，通知应用程序。

### 2. 原理：为什么它比 select 快？

* **红黑树（RB-Tree）管理**：`epoll` 在内核中维护了一棵红黑树，用于存储所有待监控的 FD。这使得添加、删除、查找 FD 的复杂度均为 $O(\log n)$，避免了每次调用都重复传递整个 FD 集合。
* **就绪链表（Ready List）**：当某个 FD 发生事件时，内核会通过中断回调机制将其放入一个双向链表中。进程只需要轮询这个链表，而不需要遍历几万个 FD。
* **内核与用户空间解耦**：不像 `select` 需要在每次调用时将所有监控对象从用户态拷贝到内核态，`epoll` 只有在添加或修改时才操作内核对象。

## 核心语法与 API 细节

`epoll` 体系由三个系统调用组成：

### 1. epoll_create

```c
int epoll_create(int size);

```

* **作用**：在内核中创建一个 epoll 实例。
* **参数**：`size` 自 Linux 2.6.8 后已被忽略，但必须大于 0。内核现在会动态分配内存。
* **返回值**：返回一个指向该实例的 **文件描述符**。使用完毕后需调用 `close()` 关闭。

### 2. epoll_ctl

```c
int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);

```

* **参数含义**：
* `epfd`：`epoll_create` 的返回值。
* `op`（操作类型）：
* `EPOLL_CTL_ADD`：注册新的 FD。
* `EPOLL_CTL_MOD`：修改已注册 FD 的监听事件。
* `EPOLL_CTL_DEL`：移除 FD。


* `fd`：需要监听的目标文件描述符。
* `event`：指向 `epoll_event` 结构体的指针，定义监听什么。


* **epoll_event 结构体**：
```c
struct epoll_event {
    //当新事件被epoll检测到后，会修改 events ，通过位运算即可得知发生了什么事件
    uint32_t events;   // 事件掩码：EPOLLIN (可读), EPOLLOUT (可写), EPOLLET (边缘触发)
    //使用 ev.data.fd 设置监听的对象
    epoll_data_t data; // 用户数据指针或 FD，方便就绪后识别是哪个对象
};

```



### 3. epoll_wait

```c
int epoll_wait(int epfd, struct epoll_event *events, int maxevents, int timeout);

```

* **作用**：等待 I/O 事件发生。它是阻塞的（取决于 timeout）。
* **参数**：
* `events`：**输出参数**。内核会将就绪的事件填充到这个数组里。
* `maxevents`：本次调用最多返回多少个事件。
* `timeout`：超时时间（毫秒）。`-1` 表示永久阻塞，`0` 表示立即返回。


* **返回值**：成功时返回就绪 FD 的个数；`0` 表示超时；`-1` 表示出错。

## 两种触发模式（关键差异）

这是面试和实战中的高频考点：

1. **水平触发 (LT - Level Triggered，默认)**：
只要 FD 处于就绪状态（比如缓冲区里还有没读完的数据），每次调用 `epoll_wait` 都会提醒你。**容错性高，逻辑简单。**
2. **边缘触发 (ET - Edge Triggered)**：
只有当 FD 的状态发生 **变化** 时（从不可读变可读），才通知一次。
* **要求**：必须配合 **非阻塞 I/O** 使用，并且要用 `while` 循环把数据读尽（直到返回 `EAGAIN`），否则会丢失后续通知。
* **优势**：减少了系统调用的次数，在高并发下效率更高。

## 使用逻辑与场景

### 1. 典型逻辑步骤

1. 调用 `epoll_create` 拿到“监控大厅”的钥匙。
2. 将服务器监听的 `listen_fd` 注册进去（`EPOLL_CTL_ADD`）。
3. 进入 `while(1)` 主循环，调用 `epoll_wait`。
4. 遍历返回的 `events` 数组：
* 如果是 `listen_fd` 就绪，说明有新连接，调用 `accept` 并把新 `client_fd` 加入 epoll。
* 如果是普通的 `client_fd`，说明有数据发过来，进行业务处理。



### 2. 使用场景

* **高性能 Web 服务器**（如 Nginx、Node.js）。
* **分布式存储/缓存系统**（如 Redis 的事件循环）。
* **长连接推送服务**：维护数十万个不活跃连接时，`epoll` 的优势能发挥到极致。

## 技术笔记：非阻塞 IO 与 epoll ET 模式深度解析

### 一、 文件描述符（FD）的阻塞属性

文件描述符的阻塞与非阻塞属性，直接决定了底层系统调用在资源未就绪时的执行行为。

#### 1. 行为差异表

| 系统调用 | 阻塞模式（Default） | 非阻塞模式（O_NONBLOCK） |
| --- | --- | --- |
| **read** | 若接收缓冲区为空，线程挂起直至有数据。 | 若接收缓冲区为空，立即返回 `-1`，`errno = EAGAIN`。 |
| **write** | 若发送缓冲区已满，线程挂起直至有空间。 | 若发送缓冲区已满，立即返回 `-1` 或部分写入字节数，`errno = EAGAIN`。 |
| **accept** | 若全连接队列为空，线程挂起。 | 若全连接队列为空，立即返回 `-1`，`errno = EAGAIN`。 |
| **connect** | 阻塞直至三次握手完成。 | 立即返回 `-1`，`errno = EINPROGRESS`，需通过 `epoll` 监测可写事件确认完成。 |

### 二、 epoll 的触发模式：LT vs ET

#### 1. 水平触发（Level Triggered, LT）

* **机制**：只要文件描述符对应的内核缓冲区处于就绪状态（有数据可读或有空间可写），`epoll_wait` 就会持续返回该事件。
* **容错性**：开发者可以只读取部分数据，下次调用 `epoll_wait` 仍能获取通知。

#### 2. 边缘触发（Edge Triggered, ET）

* **机制**：仅当描述符的状态发生变化时（如从无数据变为有数据），`epoll_wait` 才返回一次事件。
* **性能**：减少了 `epoll` 事件被重复触发的系统开销。

### 三、 为什么 ET 模式必须强制搭配非阻塞 IO？

#### 1. 逻辑必然性

在 ET 模式下，为了保证数据不丢失，开发者必须在单次事件触发中将缓冲区数据“抽干”。这要求使用 `while` 循环进行连续读取。

#### 2. 阻塞 IO 的风险

如果在 ET 模式下使用阻塞 `read`：

1. 程序进入 `while` 循环读取数据。
2. 当最后一次 `read` 耗尽缓冲区数据后，若再次调用 `read`，由于此时无数据，**线程将无限期挂起**。
3. 该线程无法回到 `epoll_wait` 处理其他 FD 事件，导致并发失效及程序假死。

#### 3. 非阻塞 IO 的作用

非阻塞 `read` 在数据耗尽时会返回 `EAGAIN`。这个特定的错误码是打破 `while` 循环、让线程重新回到 `epoll_wait` 的**唯一逻辑出口**。

### 四、 核心实现细节与逻辑控制

#### 1. 状态判断

在非阻塞 IO 下，必须精准处理以下 `errno`：

* **`EAGAIN` / `EWOULDBLOCK**`：表示当前动作已完成（数据读完或空间写满），应停止当前操作，等待下一次 epoll 通知。
* **`EINTR`**：系统调用被信号中断，需立即重试当前操作。

#### 2. 生命周期与资源管理

* **注册次数**：`listenfd` 与 `connfd` 在整个生命周期内通常只需执行一次 `EPOLL_CTL_ADD`。除非使用了 `EPOLLONESHOT`，否则无需重复注册。
* **隐式注销**：调用 `close(fd)` 后，内核会自动将其从所有的 epoll 实例中移除。

#### 3. 编程范式：ET 读操作模版

```cpp
// 1. 设置 FD 为非阻塞
setnonblocking(fd); 

// 2. 监听到 EPOLLIN 事件后
while (true) {
    ssize_t n = read(fd, buf, size);
    if (n > 0) {
        // 处理数据
    } else if (n == -1 && errno == EAGAIN) {
        // 数据读完，正常退出循环
        break; 
    } else if (n == -1 && errno == EINTR) {
        // 信号中断，继续尝试
        continue;
    } else if (n == 0) {
        // 对端关闭，执行本地关闭
        close(fd);
        break;
    }
}

```

## ET 模式的 EPOLLOUT


ET 模式的核心是“状态发生变化时才通知一次”。对于 `EPOLLOUT`，它只负责监控内核缓冲区状态：

#### 1. 刚刚把 Channel 注册到 Epoll，或者刚刚把事件从“不监听写”改为“监听写”时

当你调用 `epoll_ctl(..., EPOLL_CTL_ADD/MOD, ...)`，将 `EPOLLOUT` 塞进事件掩码的那一瞬间：

* 如果此时内核写缓冲区本身就是空的（可写的），这个状态从“未监听”变成了“可写且已监听”，**Epoll 会立刻给你触发一次 `EPOLLOUT`**。
* 因此平时不要监听 `EPOLLOUT`，只有发不出去了才监听

#### 2. 写缓冲区从“完全满了”变成“有空闲空间了”的那一瞬间

这是最典型的 ET 触发场景：

1. 你的服务器疯狂调用 `write` / `send`，直到把内核写缓冲区**完全塞满**。
2. 此时 `write` 返回了 `-1`，且 `errno == EAGAIN`（告诉你真的塞不下了）。
3. 随后，Linux 网络协议栈把数据成功发送到了网络上，客户端确认接收，导致**内核写缓冲区被腾出了空间**。
4. 就在缓冲区**从满变不满**的这一个交替瞬间，ET 模式会被激活，向你投递一个 `EPOLLOUT` 事件。

#### 3. 客户端连接刚刚建立成功时（针对非阻塞 connect）

虽然这在你的 `Connection`（处理已连接套接字）中不常用，但如果一个非阻塞 Socket 调用 `connect` 成功连上服务器，它也会触发一次 `EPOLLOUT`。

#### 注意

正因为 ET 模式**只通知一次**，如果你在收到 `EPOLLOUT` 时没有把数据发完，且没有再次遇到 `EAGAIN`，**Epoll 就再也不会通知你了**，这会导致剩余的数据永远卡在 `writeBuffer` 里发不出去。

因此可写回调函数要无限循环，直到写满或者写完
