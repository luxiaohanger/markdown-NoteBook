## 互联网应用概述
1.  **应用与协议**：
    *   **应用**：在端系统（主机）上运行的、用于网络通信的分布式进程（如Web、电子邮件）。
    *   **应用层协议**：定义应用进程间**通信规则**（消息类型、语法、语义、时序）的规范。它是应用的一部分，利用下层（传输层）提供的服务。

2.  **典型应用与对应协议**：
    *   电子邮件（SMTP/TCP）、远程终端（Telnet/TCP）、万维网（HTTP/TCP）、文件传输（FTP/TCP）、流媒体（RTP, RTSP）等。

3.  **应用体系结构**：
    *   **客户端-服务器（C/S）模式**：服务器是“始终在线”的守护进程，拥有固定IP；客户端按需启动，主动联系服务器。例如Web、电子邮件。
    *   **对等（P2P）模式**：无中心服务器，任意对等端可直接通信，每个对等端既是客户端也是服务器。具有自扩展性，但难以管理。例如BitTorrent、Skype。

## 进程间通讯
1. 寻址：IP + （TCP/UDP）端口号（port）
2. 层间接口携带的信息：数据（SDU）、双方的进程地址
3. 套接字socket：

>在 TCP/IP 协议栈中，应用进程通过调用操作系统提供的 Socket API 来实现跨网络的数据传输。

- OS本地管理的整数标识，包括通信的单方/双方；
- TCP socket：表示一个会话关系，双方的IP + 端口号（四元组）；
- UDP socket：本地的IP + 端口号；
- 减少层间信息冗余，便于操作系统管理 ，TCP传输socket + 数据，UDP传输 socket + 数据 + 对方地址 

> TCP、UDP 都没有提供安全保证，SSL是一种保证安全的加强版TCP

## 域名系统（DNS）
1.  **功能**：将人类可读的**域名**（如www.baidu.com）映射为机器可识别的**IP地址**，并提供负载均衡（一个域名对应多个IP）。

用户本机的DNS客户端向用户程序提供服务，负责解析域名，

2.  **设计目标**：唯一性、可扩展、分布式管理、高可用性。**不强求严格一致性**。
3.  **实现机制**：采用**层次化、分布式**的数据库系统。
    *   **层次化命名空间**：如 `cse.eecs.umich.edu`。
    *   **分布式服务器层次结构**：
        *   **根域名服务器**：全球13组，返回顶级域（TLD，top level domain）服务器的地址。
        *   **顶级域（TLD）服务器**：负责 `.com`, `.edu`, `.cn` 等顶级域。
        *   **权威域名服务器**：组织的DNS服务器，存储其管辖域内主机名与IP的权威映射。
        *   **本地域名服务器**：由ISP或机构维护，是主机发起查询的第一站，具有缓存功能。
4.  **解析过程**：主机向本地DNS服务器发起**递归查询**，本地服务器代表主机向根、TLD、权威服务器发起**迭代查询**，最终获得IP地址。
5.  **DNS记录**：资源记录（RR）格式为 `(名称, 值, 类型, TTL)`。关键类型包括：
    *   **A记录**：主机名 -> IP地址。
    *   **NS记录**：域名 -> 该域权威DNS服务器的主机名。
    *   **CNAME记录**：别名 -> 规范主机名。
6.  **性能与可靠性**：
    *   **缓存**：本地DNS服务器缓存查询结果，根据TTL过期，极大减少查询延迟和上游流量。
    *   **冗余**：服务器多副本部署提高可用性。
7.  **安全威胁**：DDoS攻击、中间人攻击、DNS投毒（污染）、利用DNS进行反射放大攻击。

## 电子邮件（Email）
1.  **系统组件**：
    *   **用户代理（UA,user agent）**：邮件客户端（如Outlook）。
    *   **邮件服务器**：维护用户的**邮箱**和待发邮件的**消息队列**。
2.  **邮件交付三阶段**：
    1.  **UA -> 发送方服务器**：使用SMTP。
    2.  **发送方服务器 -> 接收方服务器**：使用SMTP。
    3.  **接收方UA -> 接收方服务器**：使用邮件访问协议（POP3/IMAP/HTTP）取回邮件。
3.  **核心协议**：
    *   **SMTP（简单邮件传输协议）**：使用TCP端口25，推协议，用于**服务器间**传递邮件。命令/响应为ASCII文本，仅支持7位ASCII码传输。本身是“尽力而为”的可靠传输。
    *   **POP3（邮局协议第3版）**：使用TCP端口110，用于**用户从服务器下载**邮件到本地。支持“下载并删除”和“下载并保留”模式。**无状态**。
    *   **IMAP（互联网邮件访问协议）**：更强大的邮件访问协议。**将所有邮件和文件夹状态保留在服务器上**，支持跨会话的用户状态管理，适合从多台设备访问同一邮箱。
4.  **邮件格式与扩展**：
    *   **RFC 822**：定义邮件基本格式，分为首部和正文，由空行分隔。
    *   **MIME（多用途互联网邮件扩展）**：扩展SMTP，使邮件可传输非ASCII文本、图片、音频、视频等多媒体内容。通过额外的首部字段（如`Content-Type`）和编码（如base64）实现。

## 文件传输协议（FTP）
1.  **功能**：在异构系统间传输文件。使用TCP，端口21用于控制，端口20用于数据。
2.  **关键特性**：使用**独立的控制连接和数据连接**。
    *   **控制连接**：在整个会话期间保持，用于传输命令（如`LIST`, `RETR`）和响应。
    *   **数据连接**：每传输一个文件就新建一个，传输完毕后关闭。服务器在端口20上主动打开到客户端的数据连接。
3.  **优点**：控制“带外”（Out-of-band），服务器可维护用户状态（如当前目录、认证信息）。

## 万维网（WWW）与HTTP
1.  **基础**：
    *   **URL（统一资源定位符）**：格式为 `<协议>://<主机>:<端口>/<路径>?<查询串>`。
    *   **HTTP（超文本传输协议）**：Web的应用层协议，采用**客户端-服务器**、**请求-响应**模式。
2.  **HTTP消息格式**：
    *   **请求消息**：包含请求行（方法、URL、版本）、首部行、空行、实体主体。
        *   **方法**：GET（获取）、POST（提交数据）、HEAD（类似GET但只返回首部）、PUT（上传）、DELETE（删除）。
    *   **响应消息**：包含状态行（版本、状态码、短语）、首部行、空行、实体主体。
        *   **状态码**：如200 OK（成功）、404 Not Found（未找到）。
3.  **无状态性与Cookie**：
    *   **无状态**：HTTP服务器不保存客户端之前的请求信息。优点是可扩展、故障处理简单。
    *   **Cookie**：在**无状态协议上维护状态**的技术。服务器通过响应首部的`Set-cookie`在客户端浏览器中设置一小段数据，客户端后续请求会自动携带此Cookie，用于标识用户、实现购物车、会话等。
4.  **Web性能优化**：
    *   **问题根源**：一个网页包含多个对象（HTML、图片等），若串行获取，延迟大。
    *   **性能指标**：**响应时间 = 2×RTT + 文件传输时间**（两次RTT分别用于建立TCP连接和发送HTTP请求）。
    *   **优化技术**：
        1.  **并行连接**：同时打开多个TCP连接获取对象。
        2.  **持久连接（HTTP/1.1默认）**：复用同一个TCP连接传输多个对象，减少建立/拆除连接的开销。
        3.  **管线化**：在持久连接上，客户端可以连续发送多个请求而不必等待响应，减少等待时间。
        4.  **缓存**：将副本存储于离用户更近的地方。
            *   **机制**：通过请求头`If-Modified-Since`和响应头`Expires`、`Cache-control`控制。
            *   **位置**：
                *   **浏览器缓存**：在客户端。
                *   **正向代理缓存**：通常由ISP或企业部署，靠近**客户端群体**，减少出口流量和延迟。
                *   **反向代理缓存**：通常由内容提供商部署，靠近**服务器**，减轻源站负载。
                *   **内容分发网络（CDN）**：分布式缓存系统的规模化应用。

## 内容分发网络（CDN）
1.  **动机**：解决从单一源服务器流式传输大文件（如视频）时的扩展性、延迟和抗DDoS攻击问题。
2.  **原理**：CDN作为内容提供商的客户，在全球部署大量缓存服务器（分发节点），将内容复制到靠近用户的网络边缘。
3.  **关键技术**：
    *   **DNS重定向**：权威DNS服务器根据用户IP，将其解析到**地理/网络位置最近**的CDN服务器IP，而非源站IP。
    *   **URL重写**：将原始页面中的资源链接替换为指向CDN服务器的URL。
4.  **操作流程**：用户请求资源 -> DNS解析返回最近CDN节点IP -> 若CDN节点有缓存（命中），则直接返回；若无（未命中），CDN节点向源站或上层CDN拉取内容并缓存，再返回给用户。

## 域名解析
拆解 `xk.nju.edu.cn` 

### 1. 域名层次结构拆解
在 DNS 的分布式数据库中，域名是按照**从右向左**、由高到低的树状结构组织的：

* **根域 (Root Domain):** 在 `cn` 之后其实隐藏了一个点 `.`（即 `xk.nju.edu.cn.`），它代表 DNS 树的根。
* **顶级域名 (TLD, Top-Level Domain):** `.cn`。这是国家代码顶级域名（ccTLD），由中国互联网络信息中心（CNNIC）管理。
* **二级域名 (SLD, Second-Level Domain):** `.edu.cn`。在 `cn` 下专为教育机构开设的类别域名。
* **三级域名 (Third-Level Domain):** `nju.edu.cn`。这是南京大学（Nanjing University）在教育网下申请的专属域名。
* **四级域名 (Fourth-Level Domain):** `xk`。通常作为**主机名（Hostname）**，在校内可能对应“选课系统”（Xuan Ke）的服务器集群。

### 2. 域名解析过程 (Resolution Process)
当你通过浏览器访问该域名时，会触发以下计网核心流程：

1.  **递归查询 (Recursive Query):** 你的终端向**本地 DNS 服务器 (LDNS)** 发起请求，LDNS 负责替你“跑腿”。
2.  **迭代查询 (Iterative Query):** LDNS 会按照以下顺序跳转：
    * 询问 **根域名服务器**：得到 `.cn` 权威服务器的地址。
    * 询问 **.cn 权威服务器**：得到 `.edu.cn` 服务器的地址。
    * 询问 **.edu.cn 权威服务器**：得到南京大学 `nju.edu.cn` 权威服务器的地址。
    * 询问 **南京大学权威服务器**：最终获取 `xk` 对应主机的 **IP 地址**。


### 3. 应用层协议关联
* **端口与传输层:** DNS 解析过程通常使用 **UDP 53** 端口，因为其报文短小且对延迟敏感；但在进行区域传输（Zone Transfer）或报文超过 512 字节时，会切换到 **TCP 53**。
* **资源记录 (Resource Record):** 在南京大学的 DNS 数据库中，`xk` 对应的记录类型通常是 **A 记录**（映射到 IPv4）或 **AAAA 记录**（映射到 IPv6），也可能是一个 **CNAME**（别名记录），指向负载均衡器的域名。


> **知识点补充：**
> 为什么 `xk` 在最左边？在 DNS 的命名空间中，越靠近左侧代表其管辖范围越具体（Specific），越靠近右侧代表越通用（Generic）。这与 IP 地址的编址思想（网络号+主机号）在逻辑上是高度一致的。

## 进程 socket 端口
在计算机网络编程中，理解 **进程 (Process)**、**Socket** 与 **端口 (Port)** 之间的映射比例，本质上是在理解操作系统如何管理**并发连接**。

### 1. 进程与 Socket：1 : N
**关系描述：一个进程可以拥有多个 Socket。**

* **技术细节：** 在 Linux 系统中，“万物皆文件”。每当进程创建一个 Socket，操作系统就会在进程的**文件描述符表 (File Descriptor Table)** 中分配一个条目（例如 `fd: 3`）。
* **实际场景：** * 当你打开一个**浏览器进程**时，它会为了并行加载图片、脚本和 CSS，同时创建 6-10 个 Socket 连接到同一个服务器。
    * 一个 **高性能 Web 服务器（如 Nginx）** 会维护成千上万个 Socket，每个 Socket 代表一个活跃的用户连接。

### 2. 端口与 Socket：1 : N（最关键的理解点）
**关系描述：一个逻辑端口可以关联多个 Socket。**

这是最容易产生误解的地方。很多人认为“端口被占用”意味着其他连接进不来，但实际上：
* **监听 Socket (Listening Socket)：** 服务器进程（如 Web Server）会在 80 端口创建一个监听 Socket。
* **已连接 Socket (Connected Socket)：** 当新客户端连接时，操作系统会“克隆”或创建一个新的 Socket 来处理该特定客户端的通信，而原来的监听 Socket 继续在 80 端口等待下一个新人。
* **区分逻辑：** 操作系统通过 **“五元组” (5-tuple)** 来区分不同的 Socket，即使它们的本地端口相同：
    $$\{源IP, 源端口, 目的IP, 目的端口, 协议\}$$
    只要这五个元素中有一个不同，操作系统就能准确地将数据包分发到正确的 Socket。

### 3. 端口与 进程：N : 1
**关系描述：一个端口通常只能被一个进程“监听”，但一个进程可以监听多个端口。**

* **独占性：** 在同一台机器上，如果进程 A 已经通过 `bind()` 占用了 TCP 8888 端口进行监听，进程 B 尝试监听同一个端口时会报错 `Address already in use`。
* **多端口监听：** 一个复杂的服务器进程可以同时打开多个端口。例如：
    * 一个 **Redis 进程** 可以同时监听 6379（数据服务）和 16379（集群总线端口）。
    * 一个 **Java Spring Boot 应用** 可以同时监听 8080（业务接口）和 8081（管理监控接口）。

> 进程是发起通信的执行主体，通过内核提供的 Socket 接口（文件描述符）实现网络操作，并将该接口绑定至特定的传输层 端口，从而在 IP 地址的基础上实现网络数据的逻辑分发与多任务并发。
