## 查询顺序
### 1. SQL 的“两副面孔”

我们将这两者进行对比：

| 步骤 | **书写顺序 (Syntax)** | **执行顺序 (Logical Processing)** |
| :--- | :--- | :--- |
| 1 | `SELECT` (我要看什么) | **`FROM`** (去哪张表拿数据) |
| 2 | `FROM` (从哪拿) | **`ON` / `JOIN`** (怎么关联表) |
| 3 | `WHERE` (初步筛选) | **`WHERE`** (过滤掉不合格的行) |
| 4 | `GROUP BY` (分组) | **`GROUP BY`** (把剩下的行打包成组) |
| 5 | `HAVING` (组后筛选) | **`HAVING`** (过滤掉不合格的组) |
| 6 | `ORDER BY` (排序) | **`SELECT`** (把最终结果投射出来) |
| 7 | `LIMIT / OFFSET` (分页) | **`DISTINCT`** (去重) |
| 8 | - | **`ORDER BY`** (最后排序) |
| 9 | - | **`LIMIT`** (最后切片取几行) |



### 2. 深度拆解执行流水线

我们可以把 SQL 执行想象成一个**层层过滤的漏斗**：

#### 第一阶段：数据源 (FROM & JOIN)
这是流水线的起点。SQL 引擎先找到 `FROM` 后面的表。如果是多表连接，它会先做一个大拼表（笛卡尔积），然后根据 `ON` 条件裁掉不匹配的行。
> **这就是为什么给表起别名要在 `FROM` 里做**，因为这是第一步，后面所有的步骤才能认得这个“外号”。

#### 第二阶段：行级过滤 (WHERE)
在数据还没分组之前，先把不符合条件的行扔掉。
* *注意：* 此时还没进行聚合计算，所以你**不能**在 `WHERE` 里写 `AVG(Grade)`。

> **此时条件里所写的列名（属性）都是当前检测行的一个值，而不是一整列**；在where子句使用嵌套查询大多利用了这个思路，对每一行单独判断，注意在from中给table起别名来使用这一行的某个数据

#### 第三阶段：分组与统计 (GROUP BY & HAVING)
数据被切分成一个个小盒子（Group）。
* **GROUP BY**：把“上海”的放一起，“北京”的放一起。
* **聚合计算**：在每个盒子里算 `SUM` 或 `AVG`。
* **HAVING**：对这些算好的“盒子”进行筛选（比如只留平均分 > 80 的盒子）。

#### 第四阶段：确定展示内容 (SELECT & DISTINCT)
这时候 SQL 才去执行 `SELECT` 后面的东西。
* 它会把你在 `SELECT` 里起的**列别名**贴上去。
* *陷阱：* 因为 `SELECT` 在 `WHERE` 之后执行，所以你在 `SELECT` 里起的列别名，通常不能在 `WHERE` 里使用！

#### 第五阶段：收尾 (ORDER BY & LIMIT)
现在结果集已经拿到了，最后做个排序，然后根据 `LIMIT` 截取前几行发给你。

## 函数

### 1. 标量函数（Scalar Functions）
**特点**：对单行中的具体数值进行处理（如 `UPPER()`、`ROUND()`、`ABS()`、字符串截取等）。
**执行阶段**：**随用随算。**

* **在 `WHERE` 中使用**：发生在**过滤阶段**。
    例如：`WHERE UPPER(cname) = 'SMITH'`
    数据库在传送带上拿起一行，立刻对 `cname` 执行 `UPPER` 函数，算出结果后再进行比较。
* **在 `SELECT` 中使用**：发生在**投影阶段**。
    例如：`SELECT ROUND(price, 2)`
    数据库已经确定这一行要展示了，在最后把它“包装”成保留两列小数的样子。


### 2. 聚合函数（Aggregate Functions）
**特点**：对多行数据进行汇总计算（如 `SUM()`、`AVG()`、`COUNT()`）。
**执行阶段**：**`GROUP BY` 之后，`HAVING` 之前。**

这是 SQL 引擎最繁忙的时刻。当数据经过 `WHERE` 过滤并按 `GROUP BY` 丢进不同的“盒子”后，聚合函数开始对每个盒子里的所有数据进行扫描和计算。



### 3. 执行语序中的函数位置

我们可以把函数插入到之前的执行顺序模型中：

1.  **FROM / JOIN**：确定数据源。
2.  **WHERE**：执行**标量函数**（用于过滤行）。
3.  **GROUP BY**：将行分组。
4.  **聚合计算**：执行**聚合函数**（为每个组算出一个唯一值）。
5.  **HAVING**：利用**聚合函数的结果**进行组级过滤。
6.  **SELECT**：执行剩下的**标量函数**（格式化最终展示的值）。
7.  **ORDER BY**：如果排序列带函数（如 `ORDER BY LENGTH(name)`），在此阶段计算。

## 集合关系
解决诸如：查询至少选修了学号为A的同学所选的所有课程的学生学号

> 本质上是一个集合覆盖问题，候选人的选修课程包含了A所有课程，用**差集为空表示子集关系（包含关系）**，逻辑就很清楚了
```sql
SELECT
    `Sno`
FROM
    `SC` s1
WHERE
    NOT EXISTS(
        (   // A同学选修的所有课
            SELECT
                `Cno`
            FROM
                `SC`
            WHERE
                `Sno` = "A"
        )
        EXCEPT
            (   // s1这一行的同学选修的所有课
                SELECT
                    `Cno`
                FROM
                    `SC` s2
                WHERE
                    s2.`Sno` = s1.`Sno`
            )
    )
```
> 注意识别问题中的集合关系，（关键字：都、全部、所有）以及集合的元素是谁


例：位于同一个城市city中的所有顾客，都通过供应商aid去购买过商品pid

子集：city中的所有顾客
超集：通过供应商aid去购买过商品pid的顾客
元素：顾客

## scope
在 SQL 查询中，作用域（Scope）主要指**标识符（如表名、别名、列名）的可视化范围**。它遵循一套严格的“由内向外”和“从左到右”的层级逻辑。

我们可以将查询作用域拆解为以下三个核心法则：

### 1. 嵌套作用域：单面镜法则
这是处理嵌套查询（Subquery）时最重要的法则。
* **向内可见**：内部查询可以访问外部查询定义的属性。
* **向外不可见**：外部查询无法看到内部查询的内部细节。



> **例子：**
> ```sql
> SELECT * FROM customers c  -- 外部定义别名 c
> WHERE EXISTS (
>     SELECT * FROM orders o
>     WHERE o.cid = c.cid    -- 内部可以“看见”外部的 c
> );
> -- 这里如果写 SELECT o.oid 就会报错，因为外部看不见内部的 o
> ```


### 2. 别名生效法则：流水线顺序
正如我们之前讨论的执行顺序，别名的作用域是从 `FROM` 子句开始的。
* **定义点**：表别名在 `FROM` 或 `JOIN` 中定义后，对后续的所有步骤（`WHERE`、`SELECT` 等）都可见。
* **限制点**：在 `SELECT` 中定义的列别名，在 `WHERE` 中通常是**不可见**的，因为 `WHERE` 的执行早于 `SELECT`。



> **例子：**
> ```sql
> SELECT price * 0.8 AS discount_price
> FROM products
> WHERE discount_price < 100; -- 报错！因为执行 WHERE 时，discount_price 还没被“贴上”
> ```

### 3. 同级作用域：平级屏蔽
在同一个查询层级中，如果有重名的标识符，作用域遵循“就近原则”。

* **表名冲突**：如果你在 `FROM` 中使用了两次同一张表且没起别名（或起了相同的别名），作用域就会产生冲突，导致引擎无法分辨。
* **列名引用**：在 `JOIN` 查询中，如果两张表都有 `ID` 列，你必须使用 `表名.ID` 来明确指定作用域，否则会报“列名歧义（Ambiguous column name）”。

## 日期函数
### 1. 获取当前时间 (The Clock)
这些函数就像是你的表，用来抓取此时此刻。

* **`NOW()`**：返回当前的日期和时间（YYYY-MM-DD HH:MM:SS）。
* **`CURDATE()`**：只返回当前日期（YYYY-MM-DD）。
* **`CURTIME()`**：只返回当前时间（HH:MM:SS）。


### 2. 提取部分信息 (The Parts)
当你手里有一个日期，但你只想要其中的年份或月份时。

* **`YEAR(date)`**、**`MONTH(date)`**、**`DAY(date)`**：分别提取年、月、日。
* **`WEEKDAY(date)`**：返回星期几（0 = 周一，6 = 周日）。
* **`DATE(datetime)`**：从“日期时间”中剥离出“日期”部分。

### 3. 日期的加减运算 (The Calculator)
这是业务逻辑中最常考的地方（例如：查询过去 7 天的订单）。

* **`DATE_ADD(date, INTERVAL value unit)`**：增加时间。
    * *例：`DATE_ADD(NOW(), INTERVAL 1 MONTH)` (下个月的现在)*
* **`DATE_SUB(date, INTERVAL value unit)`**：减去时间。
    * *例：`DATE_SUB(CURDATE(), INTERVAL 7 DAY)` (一周前)*
* **`DATEDIFF(d1, d2)`**：计算两个日期之间的**天数差**（d1 - d2）。


### 4. 格式化与转换 (The Formatter)
为了把数据库里的日期变成人看得舒服的格式。

* **`DATE_FORMAT(date, format)`**：
    * `%Y`：四位年份 (2026)
    * `%m`：两位月份 (04)
    * `%d`：两位日期 (17)
    * *例：`DATE_FORMAT(NOW(), '%Y年%m月')` -> "2026年04月"*

### 5. 执行阶段与性能（重要提醒）

还记得我们之前讨论的“**作用域**”和“**执行阶段**”吗？

> **避坑指南**：
> 如果你在 `WHERE` 子句中对索引列使用日期函数，会发生什么？
> `WHERE YEAR(order_date) = 2026`
> 
> **阶段**：在 `WHERE` 过滤阶段执行。
> **代价**：由于你对列用了函数，数据库无法直接利用索引树，必须进行**全表扫描**（每一行都要算一遍 `YEAR`）。
> **优化**：改写成范围查询：`WHERE order_date >= '2026-01-01' AND order_date <= '2026-12-31'`。这样索引依然有效！

## 实践经验
1.注意DISTINCT的使用，很多时候结果都要去重
2.很多二元运算的对象要对等，例如 IN，集合内的元素形式要和集合外的一致
3.注意括号的使用，例如not exits (A except B)
4.如果两个表之间有不希望匹配的同名同domin列，不能用natural join
5.起别名是好习惯，但是起了以后原名就不能用了
