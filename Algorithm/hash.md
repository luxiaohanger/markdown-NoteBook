# 哈希表

哈希表的本质是一个“键值对”集合，其高效性的核心在于通过**哈希函数**将键（Key）直接映射到一个数组（通常称为“桶数组”）的某个位置（索引）上进行访问。不同的哈希表主要区别于**处理哈希冲突**（即不同键被映射到同一索引）的方式。

---

### 一、 哈希表通用原理与核心组件

1.  **哈希函数**：
    *   **原理**：将任意大小和类型的输入（键），通过一个计算过程，转化为一个固定范围内的整数值（哈希值）。这个值对桶数组长度取模后，即得到存储或查找的索引。
    *   **设计要求**：计算速度快、结果分布均匀（减少冲突）、确定性（同一个键每次计算哈希值相同）。

2.  **桶数组**：
    *   哈希表底层的基础数据结构，用于直接存储数据或存储数据链表的头节点。

3.  **负载因子**：
    *   **公式**：`负载因子 = 已有元素数量 / 桶数组长度`。
    *   **意义**：衡量哈希表的“拥挤程度”。负载因子越高，冲突概率越大，性能越可能下降。它是触发**重新哈希（Rehash）** 的关键指标。

---

### 二、 主要哈希表类型、流程与原理

根据解决冲突的方式，哈希表主要分为两大类：**链地址法** 和 **开放地址法**。

#### 类型一：链地址法哈希表（如 Java `HashMap`, Go `map`）

*   **核心原理**：将发生冲突的元素，在同一个桶内组织成一个链表（或树，如红黑树）。每个桶（数组元素）是一个链表头指针。

*   **数据结构**：
    ```plaintext
    桶数组: [0] -> [ (K1, V1) ] -> [ (K2, V2) ] -> ... -> null
            [1] -> null
            [2] -> [ (K3, V3) ] -> null
            ...
            [n] -> ...
    ```

*   **操作流程**：
    1.  **插入（Put）**:
        *   用哈希函数计算键 `K` 的哈希值，并取模得到桶索引 `i`。
        *   定位到桶数组的第 `i` 个位置（即链表头）。
        *   **遍历**该链表，检查是否已存在相同的键 `K`。
            *   如果存在，则更新其对应的值 `V`。
            *   如果不存在，则将新的键值对 `(K, V)` 作为**新节点插入到链表头部**（时间复杂度 O(1)）。
        *   检查负载因子，若超过阈值（如0.75），则触发 **Rehash**。

    2.  **查找（Get）**:
        *   计算键 `K` 的哈希值和桶索引 `i`。
        *   遍历第 `i` 个桶对应的链表，依次比较节点的键。
        *   找到则返回对应的值，未找到则返回空或特定标识。

    3.  **删除（Remove）**:
        *   类似查找，先定位到桶和链表中的目标节点。
        *   从链表中删除该节点（标准的链表节点删除操作）。

    4.  **重新哈希（Rehash）**:
        *   **触发**：当 `元素数量 > 桶数组长度 * 负载因子` 时。
        *   **流程**：
            *   创建一个新的、更大的桶数组（通常是原长度的2倍或一个质数）。
            *   遍历旧桶数组中的每个桶及其链表上的每一个键值对。
            *   基于**新的数组长度**，为每个键**重新计算**哈希索引。
            *   将它们插入到新的桶数组对应的链表中。
        *   **代价**：这是一个 **O(n)** 的重操作，会暂时影响性能，但能显著降低后续操作的冲突概率，保持长期高效。

*   **优点**：实现简单；有效处理冲突；负载因子可以较高（>1）；节点存储动态，无空间浪费。
*   **缺点**：需要额外的指针存储空间（内存开销）；链表过长时遍历性能下降（在JDK8的HashMap中，链表过长会转为红黑树以优化）。

#### 类型二：开放地址法哈希表（如 Python `dict` 的早期版本）

*   **核心原理**：所有元素都存储在桶数组本身中。当发生冲突时，按照某种预定的“探测序列”在数组中寻找下一个空闲的槽位。

*   **数据结构**：
    ```plaintext
    桶数组: [ (K1, V1) | (K2, V2) | 空 | (K3, V3) | 已删除 | 空 | ... ]
            索引: 0        1        2      3         4       5
    ```

*   **探测序列**：定义了冲突后寻找下一个位置的规则。常见方法：
    *   **线性探测**：依次检查下一个位置 `i, i+1, i+2, ...`。
    *   **二次探测**：按平方增量检查 `i, i+1^2, i+2^2, i+3^2, ...`。
    *   **双重哈希**：使用第二个哈希函数计算步长。

*   **操作流程**（以线性探测为例）:
    1.  **插入（Put）**:
        *   计算键 `K` 的初始索引 `i`。
        *   从 `i` 开始，顺序检查桶数组：
            *   如果桶为空，则直接放入 `(K, V)`。
            *   如果桶中的键与 `K` 相同，则更新其值。
            *   如果桶被占用（且键不同），则继续检查下一个位置（`i+1`）。
        *   检查负载因子，若超过阈值（通常比链地址法小，如0.7），则触发 **Rehash**。

    2.  **查找（Get）**:
        *   计算键 `K` 的初始索引 `i`。
        *   从 `i` 开始顺序检查，直到：
            *   找到键相等的桶，返回对应的值。
            *   遇到一个**空桶**，说明键不存在（这是关键！）。
            *   （注意：不能因为遇到“已删除”标记就停止，因为目标元素可能在更后面）。

    3.  **删除（Remove）**:
        *   这是一个复杂点。不能简单地将找到的桶置空，否则会切断后续探测路径，导致查找失败。
        *   标准做法是：找到目标键值对后，将其标记为“**已删除**”（用一个特殊的墓碑标记），而不是真正的空。

    4.  **重新哈希（Rehash）**:
        *   **触发**：负载因子达到阈值。因为开放地址法对聚集更敏感，阈值通常设得更低。
        *   **流程**：与链地址法类似，但更简单。直接创建一个新数组，遍历旧数组，将所有**有效**的键值对（跳过“空”和“已删除”）重新计算哈希并插入到新数组中。这个过程能消除所有“已删除”标记，整理碎片。

*   **优点**：所有数据存储在连续数组中，**缓存局部性好**，访问速度快；无需存储指针，空间利用率可能更高（在负载因子低时）。
*   **缺点**：实现更复杂；对负载因子敏感，必须保持较低（通常<0.7）以保证性能；删除操作需要特殊处理；容易发生“**聚集**”现象（尤其是线性探测），即冲突元素形成长串，严重降低效率。

---

### 三、 总结与对比

| 特性 | 链地址法 | 开放地址法 |
| :--- | :--- | :--- |
| **冲突解决** | 链表（或树） | 在数组中探测新位置 |
| **存储位置** | 桶数组存储链表头，数据在堆上 | 所有数据在桶数组内 |
| **负载因子** | 可较高（如0.75），甚至>1 | 通常较低（如0.7），必须<1 |
| **删除操作** | 简单，链表节点删除 | 复杂，需标记“已删除” |
| **空间开销** | 较大（需存储指针） | 较小（但需预留空位） |
| **缓存友好性** | 较差（节点内存不连续） | **好**（数据连续存储） |
| **关键优势** | 处理冲突简单，可承受高负载 | 访问速度快，内存紧凑 |
| **典型代表** | Java HashMap, C++ `std::unordered_map` | Python `dict` (早期)，Redis Hash |

根据**数据特性、内存约束和性能要求**，选择最合适的哈希表实现，或深入排查与之相关的性能问题。
